# Release Status Matrix

本文档用来防止把当前 0.4.x 公开预览版误判成最终 v1.0 发布版，也防止把当前对象/工作流/账本雏形误判成终局 v3.0。每一项都按“需求、当前证据、状态、下一步”记录。

终局 v3.0 是 **MarginNote Knowledge Agent OS**，并且必须是双模式产品：`Chat Mode` 负责轻量阅读对话，`Agent Workspace Mode` 负责对象操作、真实脑图工作台、Card Factory、Workflow Builder、Operation Ledger Explorer、Knowledge Graph Studio 和 Skill Center。当前 0.4.x 是 Chat Mode + Agent Workspace 雏形；它可以发布为公开预览版，但不能因为出现对象区、工作流区和 ledger 面板就宣称完成终局。

## 完成度矩阵

对象浏览器补充：`objectRegistryScanButton` / `扫描 MN` 已进入对象区，点击后会调用 `request_mn_object_registry_scan`，向 MN4 原生队列发送 `scan_mn_objects`，并把 `mnObjectRegistryScanFinished` 中的对象以 `native_object_scan` 证据写入 `MNObject Registry`。`objectBrowserTypeFilterSelect`、`objectBrowserKindFilterInput`、`objectBrowserSearchInput` 和 `objectBrowserFilterButton` 已接入 `object_browser` 的 `objectTypeFilter/kindFilter/query`，可按 Registry、图谱、活动、账本、kind 和关键词缩小当前对象清单。扫描对象会进入 Object Graph，显示为 `mn_note` 节点，并按 sourceRef.parentNoteId/nodePath 生成 `native_object_scan 父子边`；点击扫描对象会打开该对象图谱，点击扫描对象会打开该对象活动和账本，并把 registry objectRef 作为 `mnObject` payload 透传给 `object_graph`、`object_activity` 和 `operation_ledger_list`。这让当前版本从被动聚合已见对象，推进到可主动请求 MN4 扫描并筛选当前 notebook 可见对象；完整实时全库 Finder 式浏览仍属于后续目标。

Card Factory 补充：`generate_card` 已从普通短卡拆分推进到 `codex.mn.cardFactory.v1` 第一阶段。每张卡带 `cardType`、`source`、`learningGoal`、`reviewPrompt` 和 `codex.mn.cardFactoryCard.v1`，草稿摘要保留 `card_factory` / `card_quality`，AI 编辑确认面板显示“卡片工厂”、卡型、缺来源、长卡和重复标题，并可调用 `review_queue_add` 加入对象级复习队列。知识区通过 `review_queue_list` 显示 `codex.mn.reviewQueue.v1` 摘要和最近复习卡。跨对象去重和替换/合并仍属于后续目标。

Operation Compiler 补充：`agent_plan` 已从普通下一步建议推进到结构化操作编译第一阶段。后端现在返回 `codex.mn.operationPlan.v1`、`codex.mn.verificationPlan.v1` 和 `codex.mn.operationCompiler.v1`，把当前意图拆成计划步骤、写入步骤、required capabilities、confirmation points、rollback strategy、预期 native events 和 verification obligations。前端操作区新增 `operationCompilerPanel/operationCompilerSummary/operationPlanStats/operationCompilerChecks`，用状态色显示 schema、上下文、权限、dry-run、confirmation、rollback 和 verification 检查。`agent_plan` 会在草稿生成前对 workflow 写入步骤执行 capability-aware dry-run；当设置强制 native backend 且 `nativeCards/nativeMindmap/nativeHighlightSelection` 等必要能力缺失时，计划会直接 blocked，不再等到最终写入才报错。doctor、Web 静态检查和单文档验收都把这些控件纳入必需项。当前仍属于第一阶段编译层，不能等同于完整逐节点 native dry-run、跨对象残留扫描和发布级回滚证明。

## 当前阻塞项

原生可见高亮仍没有完整发布证据。当前可证明的是：高亮危险写库默认禁用，`diagnose_highlights` 只读诊断在权限不足时返回清楚说明，不修改原 PDF；`export_annotated_pdf` 已能生成带 PDF highlight annotation 的副本，但这不是 MN4 原生高亮。单文档全流程验收脚本已完成，但还没有针对同一篇 MarginNote 文档生成 PASS 证据。

## 发布判定

当前 0.4.28 是公开预览版发布候选，但仍不能作为最终 v1.0 发布版，因为以下硬要求还未满足：

- 原文处还没有完成带活跃选区的可靠可见 native highlighter 验证。
- 同一 topic/book 的完整按钮与工作流验收证据尚未生成。
- `release_sha256_manifest` 已通过；后续发布不再被缺 `.pkg` entry 阻塞。
- 没有跨机器安装验证。
- 已有 unsigned pkg builder、自动签名入口和 notarization 工具，但当前机器没有 Developer ID Installer 证书，也没有 notarytool 凭证，还没有真实签名/notarized pkg 或图形安装器实测。
